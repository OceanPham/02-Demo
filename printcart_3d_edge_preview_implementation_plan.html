1. Tóm tắt hướng triển khai

Hướng chính xác theo spec: xây Printcart 3D Preview API với feature đầu tiên là 3D Edge Preview cho canvas/wall-art. MVP nhận artwork/design 2D + geometry từ Product/Side/Design, render thành ảnh preview WebP/PNG, cache lại và trả về preview URL.

Đây không phải flow “upload ảnh bất kỳ → tạo object 3D”. Với canvas/wall-art, phần được 3D hóa là tấm canvas có độ dày cạnh. Ảnh artwork 2D được dùng làm texture cho mặt trước; 4 cạnh được sinh theo edge_style.

Trong Phase 1, merchant không cần upload GLB/GLTF. GLB/GLTF, mesh_name, uv_rect và model library thuộc Phase 2.

MVP: edge_preview Output: preview.webp / preview.png Backend: Laravel Renderer: Node.js + Sharp + Three.js Storage: S3/CDN No GLB in Phase 1

3. Kiến trúc end-to-end

Dashboard / Designer / WordPress Plugin ↓ Laravel backend-rest-api - auth / permission - Product / Side / Design source of truth - product_3d_profiles - design_3d_previews cache - render orchestration ↓ internal auth Node 3D Render Service - Sharp texture generation - procedural canvas geometry - Three.js scene snapshot - Playwright/Chromium headless nếu cần ↓ S3 / CDN ↓ API response: preview image URL

Luồng runtime chuẩn

  1. Merchant bật 3D Preview trong dashboard cho product.
  2. Dashboard lưu product_3d_profiles.
  3. Designer/WP/Dashboard có design_id hoặc artwork_url.
  4. Client gọi POST /v1/3d/edge-preview hoặc POST /v1/designs/{design}/generate/3d-preview.
  5. Laravel validate auth, store ownership, product/side/design relationship.
  6. Laravel resolve geometry từ Side hoặc Design.original_side.
  7. Laravel build input_hash/config_hash, check design_3d_previews.
  8. Nếu cache hit, trả result_url.
  9. Nếu cache miss, gọi Node render service bằng internal token.
  10. Node tạo front/edge textures, render procedural canvas snapshot.
  11. Laravel lưu kết quả vào S3/Image record và update design_3d_previews.
  12. Client nhận URL và hiển thị ảnh preview.
Lưu ý: react-three-fiber chỉ hợp cho interactive browser viewer. Nếu chọn server-side static render thì Node render service dùng three.js trực tiếp + Sharp + Playwright/Chromium, không dùng r3f làm core render service.

5

Laravel Render Endpoint: POST /v1/3d/edge-preview

Endpoint public/internal-facing để render một artwork/design thành static 3D edge preview.

Input

  • product_id
  • side_id
  • design_id optional
  • artwork_url optional nếu không dùng design id
  • edge_style, edge_color optional override
  • output.format, output.width, output.height
  • reset để bỏ cache

Output

  • Cache hit: status=ready, url, cache_hit=true
  • Render success: status=ready, url, cache_hit=false
  • Async: status=queued, preview_id
  • Error: validation/permission/render error rõ

Laravel orchestration

  1. Validate auth: basic hoặc unauth.token.
  2. Validate product/side/design thuộc đúng store.
  3. Resolve Product3DProfile; nếu disabled thì reject.
  4. Resolve artwork: ưu tiên design record hoặc Printcart-owned image/signed token.
  5. Resolve geometry: Side hoặc Design.original_side nếu design đã lưu.
  6. Resolve edge_style: request override → profile default → fallback.
  7. Build normalized renderer payload.
  8. Build input_hash/config_hash.
  9. Check design_3d_previews cache nếu reset=false.
  10. Gọi Node render service nếu cache miss.
  11. Lưu result vào S3/Image/cache.
  12. Return response cho caller.

Acceptance criteria

  • Render từ product_id + side_id + artwork_url chạy được.
  • Render từ design_id chạy được nếu design có artwork.
  • Cache hit không gọi Node render service lại.
  • Không accept arbitrary internal S3 path.

Cần team confirm

Endpoint này nên support artwork_url raw không? Khuyến nghị: chỉ cho URL từ Printcart CDN hoặc signed fetch token.

Renderer Step 1: fetch artwork & generate front/edge textures

Renderer cần deterministic texture generation trước khi dựng 3D scene.

Input

  • artwork.url
  • side.side_image_size
  • side.design_area
  • side.cut_line_margin
  • side.safe_zone_margin
  • profile.edge_style, edge_color

Output

  • frontTexture
  • leftEdgeTexture
  • rightEdgeTexture
  • topEdgeTexture
  • bottomEdgeTexture
  • Texture metadata: width, height, crop area

Edge style behavior

edge_styleCách sinh texture cạnhMức độ ưu tiên
solid_colorFill 4 cạnh bằng edge_colorMust-have MVP
wrapLấy strip pixel sát mép artwork và kéo qua cạnhMust-have MVP nếu làm canvas thật
mirrorLấy strip pixel mép và flip/mirrorNice-to-have MVP
blurStretch strip pixel mép và blurNice-to-have MVP
transparent/customCần định nghĩa riêngTeam confirm

Acceptance criteria

  • Texture output không méo tỷ lệ mặt trước.
  • solid_color hoạt động ổn với hex color.
  • wrap không bị đảo hướng top/bottom/left/right.
  • Renderer reject ảnh quá lớn hoặc unsupported format.

Cần team confirm

wrap có lấy vùng từ full artwork hay từ design_area sau khi compose? Khuyến nghị: render theo output artwork đã compose đúng side/design geometry, sau đó lấy strip từ artwork output cuối.

7

Renderer Step 2: procedural 3D canvas geometry & snapshot

Dựng một khối canvas procedural, apply textures vào từng mặt và snapshot ra WebP/PNG.

Input

  • Textures từ Step 6
  • width, height từ side_image_size
  • edge_depth, edge_depth_unit
  • output.width, output.height, format
  • Camera/light default

Output

  • Rendered image buffer/file
  • metadata.duration_ms
  • renderer_version
  • Optional: camera/render metadata JSON

Geometry rule

canvasGeometry:
- scene_width  = normalized(side_image_size.width)
- scene_height = normalized(side_image_size.height)
- scene_depth  = normalized(edge_depth converted to same unit)

materials:
- front  = artwork/front texture
- left   = left edge texture
- right  = right edge texture
- top    = top edge texture
- bottom = bottom edge texture
- back   = neutral material

Acceptance criteria

  • Ảnh preview nhìn rõ mặt trước và cạnh.
  • Đổi edge_depth làm độ dày preview thay đổi.
  • Đổi width/height làm ratio preview thay đổi đúng.
  • Output đúng format/size.
  • Render p95 mục tiêu 3–5 giây cho first render ở size MVP.

Cần team confirm

Góc camera default. Option A: front 3/4 view thấy cạnh phải + dưới. Option B: front chính diện ít depth hơn. Khuyến nghị: 3/4 view cho preview marketing.

Output

Cache key nên gồm

  • Design image URL/hash

  • Side ID
  • product_3d_profiles.config_version
  • Side mapping version nếu có

Acceptance criteria

Cần team confirm

  • Optional render token/preview URL lưu vào cart/order metadata
  • No local GLTF folder dependency for new flow

Integration rules

  • Không để WordPress plugin tự quyết cache key.
  • Không để plugin validate GLB/GLTF trong Phase 1.
  • Không expose internal storage path cho plugin.
  • Lazy-load widget/preview UI nếu sau này có interactive viewer.

  • Storefront/customizer gọi được API config/render bằng token hợp lệ.
  • Không có CORS issue với artwork/preview URL do dùng Printcart storage/CDN.
  • Legacy setting vẫn không phá flow cũ trong giai đoạn transition.

10. Final recommendation

Build as: Printcart 3D Preview API
First commercial feature: 3D Edge Preview for print products
Do not start with: generic 3D configurator or arbitrary product-photo-to-3D.

Điểm mạnh của Printcart là đã có print-specific data trong Side, Design, Project và preview/PDF pipeline. Vì vậy, path thực tế nhất là biến data đó thành preview 3D đáng tin cậy trước, sau đó mới mở rộng sang GLB/GLTF model library và external/commercial API.

Document này không chỉnh sửa file handoff gốc. Đây là plan triển khai bổ sung để team đọc, confirm option và chia task.
